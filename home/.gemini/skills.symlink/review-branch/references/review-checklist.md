# コードレビュー チェックリスト

## 全般

- [ ] **規約遵守**: プロジェクトのコーディング規約、命名規則、ディレクトリ構成に従っているか？
- [ ] **可読性・意図**: コードは他の開発者が読んでも意図（「なぜそう書いたか」）が容易に理解できるか？変数名、関数名は適切か？コメントは必要十分か？
- [ ] **マジックナンバー**: 意味不明な数値や文字列がハードコードされていないか？（定数やEnumに切り出されているか）
- [ ] **DRY原則**: コードの重複は避けられているか？共通化できるロジックはないか？ただし、共通化によって過度に複雑にすることは避けること
- [ ] **関心の分離**: 1つの関数やクラスが多くの責務を持ちすぎていないか？副作用（状態の変更）が予測可能か？
- [ ] **エラーハンドリング**: 想定される例外・エラーが適切に捕捉されているか？ユーザーへの通知やフォールバック（代替処理）は考慮されているか？
- [ ] **境界値と条件分岐**: `if` 文の条件は適切か？（以上の値、未満の値、null/undefined などの境界値でバグが起きないか）
- [ ] **不要なコードの削除**: デバッグ用のコード（`console.log` やプリント文）、使用していないインポート、不要な `TODO` コメントが残っていないか？

## セキュリティ

- [ ] **入力値検証**: 外部からの入力（リクエストボディ、パラメータなど）の型・長さ・形式がチェックされているか？
- [ ] **脆弱性対策**: SQLインジェクション、XSS、CSRF、OSコマンドインジェクションなどの対策は万全か？
- [ ] **機密情報の隠蔽**: パスワード、APIキー、個人情報（PII）などがログに出力されたり、ソースコードにハードコードされたりしていないか？

## パフォーマンス

- [ ] **非効率なループ・処理**: ループのネストによる計算量の肥大化（O(n²) など）はないか？ループ内で重い処理（API呼出など）を繰り返していないか？
- [ ] **データベース**: N+1問題は発生していないか？インデックスが効くクエリになっているか？不要な大量データの全件取得をしていないか？
- [ ] **リソース管理**: メモリリークの懸念はないか？（ファイル、ストリーム、データベース接続などのクローズ漏れがないか）
- [ ]

## テスト

- [ ] **テストの網羅性**: 変更・追加されたロジックに対するテストはあるか？正常系だけでなく、異常系や境界値のテストも含まれているか？
- [ ] **テストの独立性と堅牢性**: 他のテストや実行順序、実行環境（現在時刻や特定のDBデータなど）に依存せず、常に同じ結果になるか？

## 運用・変更の安全性

- [ ] **ログの適切性**: 障害発生時に原因究明ができるレベルのログ（エラー内容、文脈情報）が出力されているか？（逆に、不要なログでログ容量を圧迫しないか）
- [ ] **後方互換性**: データベースのマイグレーションやAPIの仕様変更がある場合、旧バージョンとの互換性は保たれているか？（ステージング/本番デプロイ時にシステムが止まらないか）
- [ ] **環境依存**: 環境変数（`.env` など）の追加・変更が必要な場合、その手順や設定が共有されているか？

## Git・プロセス

- [ ] **PRの粒度**: Pull Request（Merge Request）のサイズは適切か？（レビューしきれないほど巨大になっていないか）
- [ ] **コミット・差分**: 不要なファイルの修正（意図しないフォーマッタの適用など）が混入していないか？コミットメッセージは明確か？
