# コードレビュー チェックリスト

## 全般

-   [ ] **規約遵守**: プロジェクトの `GEMINI.md` に記載されたコーディング規約、命名規則、コミットメッセージ形式などを遵守しているか？
-   [ ] **可読性**: コードは他の開発者が容易に理解できるか？変数名、関数名は適切か？コメントは必要十分か？
-   [ ] **DRY原則**: Don't Repeat Yourself. コードの重複は避けられているか？共通化できるロジックはないか？
-   [ ] **関心の分離**: 1つの関数やクラスが多くの責務を持ちすぎていないか？適切に責務が分割されているか？
-   [ ] **デバッグコード**: 不要な `console.log` やデバッグ用のコードが残っていないか？

## TypeScript / JavaScript

-   [ ] **型安全性**: `any` 型を不必要に使用していないか？より厳密な型定義は可能か？
-   [ ] **Null/Undefinedハンドリング**: `null` や `undefined` になる可能性のある値を安全に扱っているか？Optional Chaining (`?.`) や Nullish Coalescing (`??`) は適切に使用されているか？
-   [ ] **エラーハンドリング**: `try...catch` ブロックは適切に使用されているか？想定されるエラーはすべて捕捉されているか？エラー発生時にユーザーやシステムに与える影響は考慮されているか？
-   [ ] **非同期処理**: `Promise`, `async/await` は正しく使われているか？`Promise.all` などで並列処理できる箇所はないか？レースコンディションの可能性はないか？

## API / サーバーサイド

-   [ ] **入力値検証**: 外部からの入力（リクエストボディ、パラメータなど）はバリデーションされているか？
-   [ ] **セキュリティ**: SQLインジェクション、XSS、CSRFなどの脆弱性につながるコードはないか？機密情報（パスワード、APIキーなど）がログに出力されたり、レスポンスに含まれたりしていないか？
-   [ ] **パフォーマンス**: 非効率なループや、データベースへの過剰なクエリ（N+1問題など）はないか？インデックスは適切に使用されているか？
-   [ ] **レスポンス形式**: APIのレスポンス形式は、既存のAPIやAPI定義と一貫性が取れているか？

## テスト

-   [ ] **テスト網羅性**: 追加・変更したロジックに対するテストは十分か？正常系だけでなく、異常系のテストも考慮されているか？
-   [ ] **テストの独立性**: 各テストは他のテストに依存せず、独立して実行できるか？
