# プルリクエストを作成 スキル

このスキルは、Git の変更履歴をレビューするための手順を定義します。

## ワークフロー

1.  **ブランチ名の確認**
    *   `git branch --show-current` コマンドで現在のブランチ名を確認します。
2.  **変更概要の把握**
    *   `git log '@{u}'..HEAD` コマンドで、トラッキングしているブランチから HEAD のブランチまでのコミット履歴とメッセージを確認し、変更の全体的な意図を理解します。
        *   もし差分が無い場合、どのブランチをベースブランチとするかをユーザーに問い合わせて、`@{u}` の部分をそのブランチに置き備えます。
    *   コミットメッセージから、どのような機能追加、バグ修正、リファクタリングが行われたのかを把握します。
3.  **変更詳細の確認**
    *   `git show <commit_hash>` コマンドを使用して、コミットごとの具体的なコードの変更内容を確認します。
    *   ファイルごと、またはディレクトリごとに変更点を整理し、以下の点に注目します。
        *   コードの追加・削除
            *   新しい機能やロジックがどのように実装されているか
        *   ロジックの変更
            *   既存のコードがどのように修正されたか
        *   リファクタリング
            *   コードの構造や可読性がどのように改善されたか
4.  **ブランチ名の変更**
    *   **ブランチ名の確認** で取得したブランチ名が機能ブランチではない場合はコミットログや変更内容に基づいて `git branch -m <branch>` で現在のブランチをリネームする
5.  **レビューと総評**
    *   **ブランチ名の妥当性**
        *   ブランチ名はプロジェクトの命名規則に沿っているか。
    *   **コミットメッセージの妥当性**
        *   コミットメッセージは変更内容を簡潔かつ正確に表しているか。
        *   プロジェクトの規約（例: 日本語での記述、特定のプレフィックスの使用など）に従っているか。
    *   **設計と一貫性**
        *   変更はプロジェクト全体の設計思想やコーディング規約に沿っているか。
        *   同様の機能を持つ他の部分と実装方法に一貫性があるか。
    *   **可読性と保守性**
        *   コードは読みやすく、理解しやすいか。
        *   変数名や関数名は適切か。
        *   コメントは必要十分か。
    *   **堅牢性**
        *   エラーハンドリングは適切に行われているか。
        *   エッジケースは考慮されているか。
    *   **改善案の提示と修正の促進**
        *   レビューの過程で修正すべき点や改善可能な箇所を発見した場合は、その内容を報告に含めること。
        *   単なる指摘にとどまらず、具体的な修正案（コードスニペットなど）を提示し、ユーザーに対してコードの修正を促すこと。
        *   ユーザーが自ら修正を行うか、またはAIに修正の実行を指示するかを選択できるように提案すること。
    *   **総評**
        *   変更の良い点を具体的に褒める（例：「関心の分離が明確になり、見通しが良くなった」）。
        *   懸念点や改善提案があれば、具体的かつ建設的に指摘する。
        *   レビュー結果をまとめて、変更に対する全体的な評価を伝えます。
6.  **レビュワー向けのポイント作成**
    *   プルリクエストの説明文に記載するための、レビュワー向けのレビューポイントを作成します。
    *   変更の核心、設計判断の背景、特に注意して見てほしい点などを簡潔にまとめます。
    *   この文章はまじめで堅い文体で作成してください。
    *   Gemini によって生成されたことがわかるように、文末にフッターとして AI によって作成されたメッセージであるという文言を入れてください。
    *   実際の内容は後述の「レビュワー向けのポイントのひな型」を参考にしてください。
    *   作成したレビューポイントを、ユーザーがコピーしやすいように一時ファイルにに書き出します。
7.  **ユーザーへの報告と確認**
    *   一時ファイルに書き出したことをユーザーに報告し、そのファイルパスを提示します。
    *   エディタで開くかどうかを `ask_user` ツールで確認します。
8.  **エディタで開く**
    *   ユーザーが同意した場合、`run_shell_command` ツールで `code --wait <file_path>` を実行してファイルを開きます。
    *   `--wait` オプションを付けているため、ユーザーがエディタでファイルを保存して閉じるまで待機します。
9.  **TODO の確認と反映（ループ）**
    *   エディタが閉じられた後、次のステップに進む前に、ユーザーが一時ファイルを編集して `todo` や `@todo` などの指示を追記していないか確認するため、`read_file` ツールで一時ファイルの内容を読み込みます。
    *   ファイル内に `todo` 等の記載が見つかった場合は、その指示の意図を解釈し、一時ファイルの内容を修正・反映します。
    *   修正後、再度ユーザーにエディタで確認するかどうかを `ask_user` で問いかけ、同意があればエディタで開き直してこのステップ（TODO の確認と反映）を繰り返します。
    *   `todo` が無い、またはユーザーがこれ以上の修正を求めない状態になったら、次のステップに進みます。
10. **プルリクエスト作成の確認**
    *   一時ファイルの内容が確定した後、プルリクエストを作成するかどうかを `ask_user` ツールで確認します。
11. **プラットフォームの判定とコマンドの選択**
    *   まず、プロジェクトのコンテキスト（`AGENTS.md` など）からプラットフォーム（GitHub または GitLab）がすでに判明しているか確認し、判明している場合はそのプラットフォームを優先します。
    *   判明していない場合は、`git remote -v` コマンドを実行し、リモートリポジトリの URL から判定します。
        *   URL に `github` が含まれる場合は GitHub と判定します。
        *   URL に `gitlab` が含まれる場合は GitLab と判定します。
    *   判定結果に基づいて、使用するコマンドを選択します。
        *   **GitHub の場合:** `gh pr create --web --assignee @me --title "<title>" --body-file "<file_path>"`
        *   **GitLab の場合:** `glab mr create --push --web --title "<title>"` を実行し、ブラウザでマージリクエスト作成ページを開きます。同時に `code --wait <file_path>` を実行して一時ファイルを開き、ユーザーに本文をコピー＆ペーストしてもらいます。
12. **プルリクエスト（マージリクエスト）の作成**
    *   ユーザーが同意した場合、選択したコマンドを実行してプルリクエストまたはマージリクエストを作成します。
    *   `<title>` には、ブランチ名などから推測した適切なタイトルを設定します。
    *   `<file_path>` には、作成した一時ファイルのパスを指定します。
13. **最終報告**
    *   以上の分析結果と、作成結果（成功した場合はURL）を、親しみやすく分かりやすい言葉でユーザーに報告します。

## 一時ファイルのファイル名

一時ファイルはプロジェクトの一時ディレクトリに `write_file` ツールで書き出してください。
一時ファイルのファイル名は、セッション内で最初に作成するときは `code-review.<ランダム文字列>.md` のように作成します。
`<ランダム文字列>` の部分には `git branch --show-current` で得たブランチ名と `git log` で得た最初のログのコミットハッシュを組み合わせて、セッションごとに一意になるようにしてください。
同じセッション内では、以降同じファイル名を使いまわします。

## レビュワー向けのポイントのひな型

「レビュワー向けのポイント作成」では、以下のひな型を参考にしてください。
ただし、必要に応じて見出しを追加しても構いません。

```
## 概要

...

## 背景

...

## 変更内容

...

## 総評

...

## レビューのポイント

...

## 改善案

このプルリク自体の改善案を記載する。
もしなにもなさそうならこの節自体を削除する。

## 今後の課題 ※もしあれば

このプルリクをマージした後に課題として残る内容があれば記載する。
もしなにもなさそうならこの節自体を削除する。

## 関連リンク

関連リンクをリスト記法で記載する。
GitHub issue へのリンクもここに url で列挙する。
もしなにもなさそうならこの節自体を削除する。

---

このメッセージは AI(Gemini) が生成した文章をベースに、人間が加筆・修正したものです。
```

## プルリクエスト更新時の注意点

プルリクエストを一度作成した後に、`fixup` コミットや追加の修正を加えて内容を更新する際の、特別なルールを以下に定めます。

1.  **`HEAD` を信頼する**:
    `git log` の見た目に惑わされてはいけません。コミット履歴がどうであれ、ブランチの最新状態 (`HEAD`) が常に「正」です。プルリクエストの本文は、必ず `git diff origin/<base-branch>...HEAD` の内容を元に作成してください。`fixup` 前の古いコード状態を前提に説明を書いてはいけません。

2.  **まっさらな状態で語る**:
    PR 本文の主要なストーリーは、「この実装は、最初からこの設計を目指して、このように実装されました」という体で記述してください。レビュワーが試行錯誤の過程を追う必要がないように、最終的な成果物だけを解説します。

3.  **議論の過程は「補足」として記述する (任意)**:
    設計の背景をより豊かにするために、検討されたものの最終的に採用されなかった代替案について言及することは有益です。例えば、以下のようなセクションを設けて、簡潔に記述します。

    ```markdown
    ### 検討された他のアプローチ

    当初、〇〇という方法も検討しましたが、△△という理由から現在の□□という設計を採用しました。
    ```

    これにより、なぜ現在の設計が最適であるかの説得力が増します。ただし、これはあくまで補足情報であり、メインの「変更内容」セクションを複雑にしてはいけません。
